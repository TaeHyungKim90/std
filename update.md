# 출퇴근 관리 — 수정 계획안 vs 현재 구현 & 구현 방향

> 기준 시점: 저장소 내 백엔드(`backend/app`)·프론트(`frontend/src`) 코드 기준으로 비교했습니다.

---

## 0. HR 정책 (본 프로젝트 적용 기준)

아래는 **구현 시 고정할 비즈니스 규칙**입니다.

### 0.1 반차 시간대 (고용노동부·근로기준법 체계)

- **법적 개념**: 반차(반일 휴가)는 **1일 소정근로시간의 절반**에 해당하는 근로시간을 사용하는 형태로 해석하는 것이 일반적입니다. 고용노동부 행정·유권해석에서도 **“하루 소정근로시간의 1/2”** 를 기준으로 안내하는 경우가 많으며, **전국 통일된 시·분(예: 반드시 12:00까지 등)** 은 법령으로 고정되어 있지 않습니다.
- **사업장 단위 적용**: 실제 **출근·퇴근 시각 구간**은 회사가 정한 **1일 소정근로시간**(예: 09:00~18:00, 점심 휴게 1시간 등)에 맞춰 **전반 4시간 / 후반 4시간**처럼 나눕니다.
- **시스템 설계**:
  - To-Do 카테고리 `vacation_am`(오전반차) / `vacation_pm`(오후반차)와 매핑.
  - 검증·표시용 **기준 시각은 코드 상수 또는 설정(환경/DB)** 으로 둠. 예: 소정근무가 09~18·휴게 13~14라면 오전 근무 구간·오후 근무 구간을 설정에 반영한 뒤, `clock_in` / `clock_out`이 **해당 반차 유형이 요구하는 “근로 절반” 구간과 합치하는지** 조회·파생 플래그로 판단.
  - 법 개정·취업규칙 변경 시 **설정만 조정**하면 되도록 하드코딩 최소화.

### 0.2 소명·결근 확정

- **소명 제출 기한**: **없음**. 시스템이 “N일 이내 미소명 시 자동 결근” 같은 **자동 마감·에스컬레이션은 두지 않음**.
- **최종 처리**: **관리자 재량**으로 근태 기록(상태·출퇴근 시각)을 수정·확정. 이미 관리자 `PATCH` 근태 API가 있으므로, **기록 누락(소명 요망)** 은 표시용 1차 분류일 뿐, 자동으로 `ABSENT`로 굳히지 않음.

### 0.3 조회 시 반차 포함 (필수)

- 관리자·시스템 **모든 근태 조회(기간 조회·Drawer·향후 캘린더 연동)** 에서 해당 `work_date`의 **반차·연차 To-Do를 반드시 조인·병기**.
- 표시 예: `반차(오전)` + 출근 시각, `[연차] 08:50 출근`, 반차일인데 출퇴근 없음 → `기록 누락(소명 요망)` 등.
- **백엔드** `get_user_attendance_range`(및 확장 API) 응답에 `half_day_type` / `vacation_todo_summary` 등 **반차 포함 메타**를 넣는 것을 계획의 일부로 둠.

## 1. 출퇴근 관리 (관리자·시스템 화면)

### 1.1 휴가자 출근 기록 처리 (캘린더 최우선 + 병기) — **반차·연차 모두 포함**

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 캘린더에서 휴가 최우선 표기 | `UserAttendanceDrawer`는 **근태 API 행 목록**만 표시. To-Do(연차/**반차**)와 **셀 단위 병합 표시 없음**. | 동일 기간 **`Todo`(vacation_full / vacation_am / vacation_pm 등)** + `Holiday` 조회 후 **날짜 키로 조인**. **반차 행은 반드시 조회 결과에 포함**해 라벨 표시(§0.3). |
| 휴가인데 출근 기록 존재 | UI에서 동시 존재 강조 없음. | 응답에 `vacation_summary`(카테고리명·반차 오전/오후 구분) + 출근 시각 병기: 예 `[연차] 08:50 출근`, `[오전반차] 09:10 출근`. |

### 1.2 반차 사용자 검증 (결재 + 실제 출퇴근 둘 다)

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 반차일 “정상 근무” 판정 | **없음**. To-Do와 출퇴근 **교차 검증 없음**. | **조회 시점**(또는 배치)에 해당일 `vacation_am`/`vacation_pm` 존재 시: (1) 출근+퇴근 존재 여부 (2) §0.1 **설정된 소정근로 절반 구간**과 시각 비교 → `derived_flag`(예: `needs_review`, `half_day_ok`)를 응답에 포함. **최종 판정은 관리자 수정으로만 확정**(§0.2). |

### 1.3 반차자 미출근 → 「기록 누락(소명 요망)」 (자동 결근 없음)

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 미기록 분류 | 평일 등에서 Attendance 없으면 **`ABSENT` 가상 행**. 반차 전용 1차 상태 없음. | **반차 To-Do가 있는 날** + 출퇴근 없음 → 가상행은 **`ABSENT` 대신 `기록 누락(소명 요망)`**(또는 동등 코드). **소명 기한 없음**; **관리자가 재량으로** `ABSENT`·`LATE`·`NORMAL` 등으로 수정. 일반 연차일만 해당되면 기존처럼 결근 처리 제외 정책과 조합 가능. |

### 1.4 공휴일 표기 (주간 근무 조회)

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 공휴일 UI | 주말 스타일만, 공휴일 API **미연동**. | 기간 내 공휴일 API 조회 → 공휴일 열 **명시 라벨**. **반차가 공휴일과 겹치는 경우**는 드물지만, 조회 시 To-Do와 함께 표시하면 충돌 확인 가능. |

### 1.5 휴일·주말 근무 표시

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 주말/공휴일 | 가상 결근 제외, 실제 기록만 행 표시. | §0.3대로 **해당일에 반차/연차 일정이 있으면 조회 행에 반드시 반영**(주말 근무 기록이 있으면 병기). 필요 시 주간 **7일 고정 그리드**. |

## 2. 출퇴근 액션 (사용자 화면)

### 2.1 휴가 중 출근 시도 — 확인 팝업 후 허용

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 동작 | 휴가 시 **백엔드 차단**·프론트 **비활성**. | **종일 연차 등**은 확인 후 허용으로 변경. **반차**는 §2.2. |

### 2.2 반차 중 출근 — 경고 없이 정상

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 동작 | 반차 전용 분기 없음. | **유지·명시**: `vacation_am`/`vacation_pm` 일정이 있어도 **추가 확인 팝업 없이** 출퇴근 버튼 동작. 서버는 **종일 휴가**만 확인 팝업 정책과 묶어 차별화. |

### 2.3 휴일·주말 출근 — 확인 팝업

| 항목 | 현재 구현 | 구현 방향 |
|------|-----------|-----------|
| 동작 | 없음. | 휴일·주말·공휴일 확인 문구 후 허용. **반차가 평일에만** 걸리는 경우가 일반이므로 2.2와 구분. |

---

### 2.4 병가/경조휴가/공가 처리 정책 (요청 반영)

| 구분 | 계획(요구사항) | 구현 시 시스템 포인트 |
|------|------------------|-------------------------|
| 병가(`vacation_sick`) | **출근/퇴근 기록 정상 등록**(차단 팝업/비활성 없음) | `assert_user_can_clock_in`에서 병가 To-Do는 출근을 막지 않도록 예외 처리. 필요 시 조회 DTO/프론트에서 병가 라벨을 표시하도록 `Attendance.status` 또는 요약 메타 확장을 계획. |
| 경조휴가(`vacation_special`) | **병가와 동일하게 출근/퇴근 기록 정상 등록** | 병가 예외 로직을 경조휴가로 동일 적용. 급한 상황 대응을 위해 확인 팝업 없이 진행. 표시용 라벨/메타 확장 계획. |
| 공가(`official_leave`, “공가”) | 출근 시 **“출근 기록 등록하시겠습니까?” 확인 후 진행**, 그리고 **해당 일정도 업데이트** | 서버 차단을 유지한다면 프론트 확인 플로우가 필요(예: confirm 플래그/에러코드 기반 분기). clock-in/out 완료 후 공가 To-Do를 “처리완료” 형태로 업데이트하는 방식을 계획(예: `processed_at` 추가 또는 `title/description`에 처리 표시). |

## 4. 권장 구현 순서 (코딩 시)

1. **설정·상수** — 1일 소정근로시간·휴게·오전/오후 반차 구간(§0.1)을 `core/config` 또는 DB로 정의.  
2. **백엔드 조회 확장** — `get_user_attendance_range`에 **일자별 To-Do(반차 포함)** 조인, `vacation_todo_summary` / `half_day_type` / `review_hint` 필드 추가.  
3. **가상 행 규칙** — 반차일 무기록 → `기록 누락(소명 요망)`; **자동 결근 확정·소명 기한 로직은 구현하지 않음**.  
4. **백엔드 출근 정책** — 종일 휴가 vs 반차 분리, 휴가일 출근 허용 시 `assert_user_can_clock_in` 조정.  
5. **프론트** — 관리자 Drawer에 반차·연차 병기 컬럼/뱃지; 사용자 화면 확인 모달.  
6. **관리자 재량** — 기존 PATCH로 최종 상태 정리(추가 UI는 필터·일괄 표시 정도로 가볍게 가능).
